## 1. 開発・運営体制

### 1.1 チーム構成

| 役割 | 担当 | 備考 |
| --- | --- | --- |
| **代表 / プロダクトオーナー** | ヒロ | コンセプト設計、事業開発、農家・デザイナーとの提携交渉 |
| **フルスタックエンジニア** | ヒロ（+ 1名想定） | Next.js / Supabase / Stripe / Flower Story System 開発全般 |
| **フラワーデザイン / 制作** | 代表 or フリーランス契約 | 初期は自社内製（ロス花の目利き、制作、梱包、配送手配） |
| **デザイン（UI/ビジュアル）** | 初期は代表兼任 → 外注 | ロゴ・商品写真・SNS素材・コレクタブルボックスデザイン |
| **農家リレーション** | 初期は代表兼任 | Flower Story System用のコンテンツ収集（富山のネットワーク活用） |

1〜2人体制で始め、売上に連動してチームを拡大する方針。初期は「エンジニアリング」と「花の調達・制作」の両輪を、実家の卸売インフラを最大限活用して内製化する。

### 1.2 拠点（ハイブリッドモデル）

* **東京都文京区:** テック開発、マーケティング、大田市場からのテスト調達、投資家・メディアとのリレーション、東京都創業助成事業の申請拠点。
* **富山県（ヒロの実家）:** フラワーロス花の安定的な仕入れ、農家ネットワーク構築、Flower Story用コンテンツ収集、そして**初期の主要な制作・配送拠点**。

**🚨 オペレーションの基本方針:**
「ロスフラワーを救う」というストーリーの真正性を担保するため、**初期は在庫を持たないドロップシッピングや外部花屋への丸投げは行わない。** 富山の実家（卸売）のルートを活用して市場段階の余剰花（ロス）を直接調達し、自社（または専属のフリーランスデザイナー）で制作・梱包・発送・Flower Storyのデータ入力までを一貫して行う「自社制作モデル」でスタートする。

## 2. オペレーション設計

### 2.1 注文〜配送フロー（Flower Story対応版）

1. **顧客がサイトで注文 → Stripe決済完了**
2. **Webhook → Supabaseに注文保存**
3. **★ バッチ（ロット）を注文に紐づけ:** どの産地・農家のロス花を使用するかを決定。
4. **注文確認メール自動送信（顧客へ） + ★ Flower Miles付与処理**
5. **制作・梱包:** 担当デザイナー（代表またはフリーランス）が、コレクタブルボックスにアレンジメントを制作。
6. **★ Flower Timeline記録:** デザイン日・発送日をシステムに記録。
7. **ヤマト運輸にて発送（夏季はクール便必須）**
8. **配送完了通知（顧客へ） → ★ Flower Story（花/人/時間の3ストーリー）がマイページで閲覧可能に**
9. **フォローアップ（花のお手入れTips / ドライフラワー化の手順 / レビュー依頼）**
10. **★ Flower Impact更新（累計救済花数・CO2等を加算）**

### 2.2 初期のオペレーション（手動中心）

Phase 1（MVP）では全自動化は不要。ただし、Flower Story用のデータ入力は初期から組み込む。

* **注文管理:** Supabaseダッシュボードで注文を確認。ステータス更新も手動。
* **バッチ登録（★新規）:** 花の仕入れ時に、Supabaseダッシュボードでバッチ情報（産地、花の種類、農家、仕入れ日、廃棄予定日）を手動登録。初期は週1〜2回の登録で対応。
* **配送:** 富山拠点または東京拠点からヤマト運輸で直接配送。
* **問い合わせ:** メール（Resend）+ LINE公式アカウント。初期は代表が直接対応。

### 2.3 Flower Story コンテンツ制作・農家オンボーディング

**初期（ローンチ〜6ヶ月）のフロー:**

1. **アプローチ:** ヒロの実家ネットワーク経由で富山・近隣県の農家にコンタクト。
2. **説明・同意:** Bloomboxの理念（ロスフラワー活用）を説明し、Flower Storyへの掲載同意（露出レベル選択）を得る。
3. **コンテンツ収集:** 農家の自己紹介テキスト（50〜100字）と花/農家本人の写真を提供してもらう。
4. **登録:** 管理者がSupabaseに手動登録。

**農家の露出レベル:**

* **Level 1（地域のみ）:** 「富山県の農家」（デフォルト）
* **Level 2（名前あり）:** 「富山県砺波市 田中花子さん」（※書面での同意取得）
* **Level 3（写真あり）:** フルプロフィール + 写真 + コメント（※肖像権使用同意書取得）

### 2.4 Flower Miles運用

* **Miles付与:** Stripe Webhook `payment_intent.succeeded` 受信時に自動付与。
* **ランク更新:** 毎日深夜のVercel Cronバッチで集計・更新。LINEで通知。
* **クーポン発行:** ランクアップ時にStripe Couponを自動生成（有効期限30日）。
* **不正対策:** 注文キャンセルパターン監視。Miles付与は「配送完了後」に確定させる。

### 2.5 自動化のロードマップ

| フェーズ | 内容 | タイミング |
| --- | --- | --- |
| **Phase 1** | 注文通知/確認メール自動化、Flower Miles自動付与 | テスト販売時（5〜6月） |
| **Phase 2** | 簡易管理画面（バッチ管理/農家管理）、Flower Story自動生成 | 本格ローンチ（7〜8月） |
| **Phase 3** | 農家向け専用入力フォーム提供（写真アップロード等） | 月100件超 |
| **Phase 4** | 在庫管理・配送追跡の完全自動化、Flower Map全国対応 | 全国展開時 |

## 3. 花の調達・提携戦略

### 3.1 初期モデル：実家インフラを活用した自社調達

* **調達:** 富山県で花き卸売業を営む実家のネットワーク、および大田市場から、規格外や需給ミスマッチによる**「市場段階のロスフラワー」**を直接買い付ける。
* **制作:** ブランドイメージを体現できるフリーランスのフラワーデザイナーと業務委託契約を結び、東京または富山拠点で制作を行う（初期は代表自ら学ぶことも検討）。
* **メリット:** 中間マージンをカットし、ロスフラワーを低コストで仕入れることで、コレクタブルボックスへの投資を行いつつ高い粗利率（50〜60%）を確保する。

### 3.2 配送

* **初期:** 富山拠点または東京拠点からのヤマト運輸宅急便。
* **夏季対応:** 花の品質維持のため、6月〜9月は**クール便（冷蔵）の利用を必須**とする（※配送料金は2025年4月改定後の+330円〜+550円を想定に組み込む）。

## 4. 法務・コンプライアンス

### 4.1 事業形態

初期: **個人事業主**として開始。年商500万円〜 or 助成金採択/外部資金調達時に法人化。

### 4.2 必須の法的対応

**特定商取引法に基づく表記**

* ECサイトに販売業者名、所在地、電話番号、販売価格、送料、返品ポリシー等を明記。生花はクーリングオフ対象外（品質不良時のみ交換対応）。
* **🚨 プライバシー保護:** 代表個人の自宅住所公開を避けるため、**バーチャルオフィスの住所利用**、およびプラットフォーム（Stripe等）の住所・電話番号の代替表記制度を活用する。

**プライバシーポリシー（個人情報保護法対応）**

* 取得する個人情報と利用目的（Stripe決済への連携含む）。
* **農家・花屋の情報:** Flower Storyで表示する情報は本人の同意（露出レベル）に基づく旨を明記。
* **Miles・Impactデータ:** 購買履歴に基づく統計情報の収集と利用目的を明記。

**利用規約**

* サービスの利用条件、免責事項（天候による配送遅延・品質変動等）。
* **Flower Milesプログラム規約:** ポイント付与条件、有効期限、ランク制度。

### 4.3 決済・その他のコンプライアンス

* **決済:** Stripe Checkout使用によりPCI DSS SAQ A準拠（カード情報は非保持）。
* **景品表示法:** Flower Milesのクーポン付与は取引価額の20%以内（総付景品の上限）に収める。
* **花き・食品衛生法:** 生花EC販売に許認可は不要。ただしドライフラワーを販売する場合は内容物表記に注意。

## 5. セキュリティ

| 対策 | 内容 |
| --- | --- |
| **カード情報** | Bloomboxサーバーには一切保存しない（Stripe管理） |
| **DB アクセス制御** | Supabase RLS で顧客ごとにデータ分離 |
| **不正カード決済** | Stripe Radar（機械学習ベースの不正検知）有効化 |
| **eギフト悪用** | 推測困難なワンタイムURL（ランダムトークン）発行、有効期限設定 |
| **Flower Miles** | RLSで本人のみ閲覧・利用可能に制限、改ざん防止 |
| **DDoS攻撃** | Vercel Edge Networkの標準保護（※CloudflareはDNSのみ使用） |

## 6. 知的財産

### 6.1 商標リスクヘッジ

「Bloombox」という名称は、過去の美容系サービス等で認知があり、第31類（花）や第35類（小売）で既に類似商標が登録されている可能性が高い。

* **対策:** 事業開始前にJ-PlatPat等で厳密な商標調査を行う。登録済みの場合に備え、**「Bloom&Box」「Re:Bloom」「ロスブルーム」などの代替ブランド名候補も並行して準備**しておく。
* **出願:** 東京都創業助成事業の対象経費を活用し、早期に商標出願を行う。

### 6.2 ドメイン

bloombox.jp （または代替ブランド名の.jp / .com）を優先的に取得する。

## 7. 保険・リスク管理

### 7.1 想定リスクと対策

| リスク | 影響度 | 対策 |
| --- | --- | --- |
| 配送中の花の損傷 | 中 | 梱包のテスト（落下テスト等）、写真付きクレームでの迅速な再送対応 |
| 配送遅延 | 中 | ヤマト運輸の追跡情報連携、余裕を持った配送日指定 |
| 花の品質トラブル | 中 | 実家（卸売）のプロの目利きによる品質管理、夏季クール便の徹底 |
| 個人情報漏洩 | 高 | RLS + HTTPS + Stripe管理による最小限のデータ保持 |

### 7.2 保険

万一のアレルギーや事故に備え、**PL保険（生産物賠償責任保険）**への加入を検討（年額¥10,000〜30,000程度）。

### 8.2 重要な判断ポイント

* **本格ローンチタイミングの変更:** ロスフラワーを活用する事業モデルの特性上、市場の花が不足する「母の日（5月）」をメインローンチにするのは矛盾する。**ロスが最も発生し、かつ女性の誕生月が最も多い「夏（7〜8月）」の本格稼働を最優先とする。** 5〜6月はクラウドファンディングを通じたMVPテスト期間とする。
* **法人化タイミング:** 年商500万円超 or 助成金採択 or 外部資金調達のいずれかの段階で検討。
